### Morphological Analysis — Batch 1（基于你的约束）

_约束输入：_ 总入口命令=`菜单`；优先支持服务=`activity`/`ai`/`vision`；仅管理员可修改配置。

**[UX #1]**: 总菜单-服务一览卡
_Concept_: 复用 `src/vendors/nonebot_plugin_command_handler` 的模式：`/菜单` 触发一张卡片，列出 `activity/ai/vision` 当前状态（开/关/错误），并给出序号进入子菜单。普通成员看到只读卡（状态+说明），管理员看到可操作卡（含“进入设置”）。
_Novelty_: 把“开关配置”变成可视化入口，不依赖记忆命令；同一入口对不同权限展示不同操作。

**[IA #2]**: 子菜单-统一开关页
_Concept_: 每个服务子菜单第 1 项固定为“开/关”，第 2 项固定为“查看状态/诊断”，其余是服务特有参数。这样用户肌肉记忆稳定，减少学习成本。
_Novelty_: 用“菜单信息架构一致性”降低出错与误操作。

**[自动化 #3]**: 元数据驱动自动菜单
_Concept_: 为服务类（或 service_action/ai_tool 元数据）补一个轻量 `__service_settings__` 描述（显示名、默认开关、可配置参数 schema、帮助文案）。菜单生成器扫描这些元数据自动生成总菜单与子菜单命令列表。
_Novelty_: 让“新增服务/新增配置项”几乎不需要写菜单代码，只写元数据。

**[权限 #4]**: 权限前置+只读分流
_Concept_: `/菜单` 总入口对所有人开放，但进入“修改开关/保存配置”的 handler 前置校验管理员；非管理员直接落到只读视图，并附“联系管理员开启”的提示。
_Novelty_: 把权限当成“路由规则”，而不是散落在每个按钮/每段逻辑里。

**[存储 #5]**: 配置存储与默认值合并
_Concept_: 统一一个 group 级配置存储层（例如 `service_config_store`），读取时做三层合并：代码默认值 < 全局配置 < 群配置。菜单修改只写“群配置差异”，避免写全量。
_Novelty_: 避免配置漂移与升级痛点，且便于回滚（删除差异即可恢复默认）。

**[可见性 #6]**: 开关与命令可见性联动
_Concept_: 服务关闭时，菜单里仍显示该服务但标注“已关闭”；同时把该服务的命令入口在帮助/菜单中隐藏或置灰（而不是用户点进去才报错）。
_Novelty_: “在用户行动之前阻止错误”，减少“操作超时或出错”的主观体验。

**[可观测 #7]**: 超时统一反馈与诊断码
_Concept_: 为所有菜单 handler 增加统一的“执行包装器”：超时（比如 8s/15s）返回友好提示 + 诊断码（含 handler 名、服务名、trace id），管理员可用“诊断码查询”拿到详细日志提示。
_Novelty_: 把“超时”从黑盒变成可定位事件，且不把技术细节打扰普通用户。

**[回滚 #8]**: 一致的回滚/灰度开关
_Concept_: 新菜单系统增加一个总开关：按群启用/禁用（例如 `menu_v2_enabled`）。先在测试群开启，出问题一键关闭回到旧命令；并允许“只启用 activity/ai/vision 三个服务菜单”的灰度子开关。
_Novelty_: 菜单系统本身也可灰度，降低上线风险。

**[扩展 #9]**: 按服务扩展的参数页
_Concept_: 对 `ai/vision` 等服务提供“参数页”子菜单：例如 `tool_max_rounds`、模型选择、限流、是否启用某类工具；对 `activity` 提供“活动参数页”（比如默认持续时间、报名规则）。参数页使用统一表单渲染（字段名/说明/当前值/默认值）。
_Novelty_: 把“开关”升级为“可配置面板”，但仍保持结构统一。

**[测试 #10]**: 菜单自动化的测试基线
_Concept_: 为菜单生成器做快照测试：给定一组服务元数据与群配置，输出菜单文本/卡片数据应稳定；并增加权限用例（管理员/普通成员输出不同）。
_Novelty_: 把“菜单可用性”纳入回归，避免未来新增服务时把菜单弄坏。

### Role Playing — Batch 1（你提出的“双入口”形态）

_约束输入：_ 两个入口命令：`菜单`（管理员+群友都可用，用于调用功能）与 `设置`（仅管理员可用，用于开关与参数）。设置管开关，菜单管执行。

**[UX #11]**: 双入口一致的导航结构
_Concept_: `菜单` 与 `设置` 的首页都按同一服务顺序呈现（activity/ai/vision…），并使用同一序号进入对应服务页；只是每个服务页的可用操作不同（菜单=执行；设置=开关/参数）。
_Novelty_: 让用户“从菜单学到的导航”可迁移到设置，降低认知负担。

**[UX #12]**: 菜单页只展示“能做什么”
_Concept_: `菜单` 里每个服务只显示 3 件事：当前是否可用（开关状态）、3 个最常用功能入口、1 个示例用法；其余高级功能不在菜单堆满，而在服务子菜单里逐层展开。
_Novelty_: 以“可执行任务”而非“配置项”组织信息，减少误触发与迷路。

**[安全 #13]**: 设置页权限前置+显式说明
_Concept_: 非管理员触发 `设置` 时，直接返回“仅管理员可用”的只读说明（含如何申请权限），不进入任何 flow；避免进入 flow 后才提示导致用户等待超时。
_Novelty_: 把权限当作入口门禁，减少交互回合与超时概率。

**[安全 #14]**: 危险操作二次确认
_Concept_: `设置` 中对高风险开关（例如 vision 生成、外网调用、费用相关）做二次确认：再次输入“确认/我已了解”；并显示影响范围（本群/全局）。
_Novelty_: 通过“摩擦”降低误操作，而不是事后补救。

**[运维 #15]**: 设置变更审计与回滚按钮
_Concept_: `设置` 每次保存后记录审计（操作者、时间、变更前后差异），并提供“回滚到上一版/恢复默认”一键入口。
_Novelty_: 把回滚设计成一等公民，避免配置被改坏后只能手工修数据库。

**[产品 #16]**: 开关影响范围可视化
_Concept_: 在 `设置` 的每个开关旁显示“影响哪些命令/菜单项/工具”（例如关闭 vision 会隐藏菜单里的“识图/生图”入口），让管理员看到后果。
_Novelty_: 让“开关”从抽象变成可预期，减少反复试错。

**[系统 #17]**: 菜单与设置共享同一元数据源
_Concept_: 为每个服务声明一份元数据：displayName、enabled_key、常用功能（菜单入口）、可配置参数 schema（设置入口）。菜单与设置都从这份元数据自动生成。
_Novelty_: 真正避免“两套逻辑”，新增服务只写一次声明。

**[性能 #18]**: flow 交互超时分级
_Concept_: `菜单` 的执行型操作默认短超时（如 15s）并支持“后台继续/稍后查询结果”；`设置` 的输入等待型操作用更长超时（如 60s）并可随时“退出/返回”。
_Novelty_: 不同意图使用不同超时策略，减少“操作超时或出错”统一兜底的发生率。

**[一致性 #19]**: 菜单调用只调用“已开启能力”
_Concept_: `菜单` 里不出现已关闭服务的执行入口（或置灰并解释）；如果用户通过旧命令触发，也统一返回“请管理员在设置开启”并附快捷入口提示。
_Novelty_: 让开关与可见性联动，减少无效调用与错误噪音。

**[测试 #20]**: 菜单可用性回归用例
_Concept_: 增加回归测试矩阵：管理员/非管理员 × 服务开/关 × 入口（菜单/设置）× 超时/异常路径，断言输出文本/卡片结构稳定。
_Novelty_: 把“交互稳定性”量化，避免后续改服务把菜单玩坏。

**[架构 #21]**: 开关-指令映射表（单一真源）
_Concept_: 为每个服务维护一张“指令 -> 开关key”映射表（或由元数据自动生成）。菜单生成时按当前开关过滤指令；设置页展示开关时反向列出“影响哪些指令”。  
_Novelty_: 让“菜单可见性”和“开关影响范围”从同一份数据推导，避免两套判断逻辑漂移。

**[自动化 #22]**: 指令标记自动发现（零手工列表）
_Concept_: 扫描服务方法上的元数据（例如 `@service_action` 的 `cmd/desc/visible`）自动生成“可执行指令列表”；菜单页默认展示这些指令（可分页/搜索）。  
_Novelty_: 新增指令无需改菜单代码，天然覆盖“所有标记为指令的功能”这一需求。

**[配置 #23]**: 服务设置 schema 化（开关/参数统一描述）
_Concept_: 每个服务提供一份 `settings_schema`：包含总开关、子开关、参数项（类型/默认值/说明/是否危险）。设置页按 schema 自动渲染。  
_Novelty_: 把“配置 UI”从硬编码文本流变成数据驱动，扩展成本极低。

**[实现 #24]**: 动态生成子菜单项（按开关/权限实时变化）
_Concept_: `设置` 页对管理员：显示所有开关与参数；对非管理员：入口直接拦截。`菜单` 页对所有人：只显示当前 enabled 的指令，并标注“由设置控制”。  
_Novelty_: 菜单/设置的动态性来自同一套 schema 与开关状态，体验一致且可解释。

**[一致性 #25]**: 统一开关 guard（装饰器/中间件化）
_Concept_: 把开关判断从每个 handler 内 if/else 提升为统一装饰器（类似你们已有 `@check_enabled`）：支持“服务总开关”和“子功能开关”两级。  
_Novelty_: 让“开关逻辑”成为基础设施，避免漏判导致的超时/报错。

**[运行时 #26]**: 立即生效的配置写回策略
_Concept_: 设置页保存后立刻写入群配置并刷新内存缓存（或让 service_manager 下次读取时命中最新值）；同时给出“已生效”的确认卡。  
_Novelty_: 避免“改了但没生效”的体验问题，减少重复操作与误报超时。

**[可观测 #27]**: 配置变更事件通知（可选）
_Concept_: 保存成功后向管理员私聊/群内发一条简短审计：谁把哪些开关从 off→on（含指令影响列表）。  
_Novelty_: 让配置变更可追溯，排障时能快速关联“最近谁改过设置”。

**[复用 #28]**: 只读/可写视图复用（模板化输出）
_Concept_: `菜单` 与 `设置` 复用同一套“服务卡片渲染器”，只是在 action 区域注入不同按钮/指令列表。  
_Novelty_: 输出一致、样式一致、减少维护两套文案。

**[可用性 #29]**: 菜单分页与搜索
_Concept_: 当“所有标记为指令的功能”过多时，菜单支持分页（1/2/3）与关键字搜索（输入关键字过滤指令）。  
_Novelty_: 防止菜单变成刷屏清单，仍保持可发现性。

**[防呆 #30]**: 子功能默认继承总开关 + 显式例外
_Concept_: 约定：服务总开关 off 时子开关不生效（强制不可用）；总开关 on 时子开关逐项控制。设置页展示这一继承关系，避免管理员误解。  
_Novelty_: 用清晰的层级规则避免“开关组合爆炸”与歧义。

### Theme 2：双入口信息架构（菜单 vs 设置）
_Focus_: 让所有人能执行、管理员能配置，并且导航一致、可迁移。

**Pattern Insight:** “同一服务，同一序号/路径，不同权限不同操作面板”，减少学习成本。

### Theme 3：自动化生成与单一真源（避免两套逻辑）
_Focus_: 通过元数据/schema 让菜单/设置/可见性/开关影响范围都自动推导。

- 动态生成子菜单项

**Pattern Insight:** 把“可执行指令”“可配置项”“开关影响范围”统一到同一份描述数据，减少维护成本与漂移风险。

### Theme 4：权限、安全与回滚（可控上线）
_Focus_: 管理员可安全操作，可追溯，可一键恢复。  

- 权限前置+只读分流

**Pattern Insight:** 权限与回滚应该是系统级能力，不应散落在每个 handler 的 try/except 里。

### Theme 5：可用性与测试（规模化后不崩）
_Focus_: 指令多了也不刷屏；菜单不会因新增服务而坏。  

- 菜单分页与搜索

- 菜单可用性回归用例
