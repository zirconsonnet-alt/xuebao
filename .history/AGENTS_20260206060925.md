# AGENTS — 项目级协作约束（对所有 Codex/代理工具生效）

你在此仓库中作为代码代理/助手工作时，必须严格遵守以下规则。任何与这些规则冲突的默认行为都必须被覆盖。

## 0. 最高优先级：交流语言
- 必须始终使用中文与用户交流（包括提问、解释、计划、结论、错误信息）。
- 除非用户明确要求，否则不要切换到英文或夹杂英文长段说明。
- 代码中的标识符/库名按原样保留，不强制翻译。

## 1. 禁止的代码改动行为（需要先获批）
- 在用户明确说“可以改”或“按你说的方案改”之前：
  - 不要直接修改任何文件；
  - 不要提交补丁、diff、PR；
  - 不要运行会产生写入的格式化/重构/自动修复；
  - 不要“顺手”做你认为显而易见的小修小改。
- 允许做的：只读分析、提出修改方案、给出候选 diff（但不得落盘应用），以及列出需要改动的文件与具体位置。

## 2. Python 代码硬约束
- 绝对不要在任何 Python 文件中添加或保留：
  - `from __future__ import annotations`
- 如果发现已有上述导入：
  - 先提示用户它存在于哪些文件/位置；
  - 给出移除后的兼容性影响评估；
  - 等用户确认后再执行移除。

## 3. 工作流（强制）
- 每次接到需求后，按以下顺序输出：
  1) 你对需求的复述（中文，尽量精确）
  2) 影响范围：可能涉及的模块/文件清单（只读推断，不要改）
  3) 修改方案：分步骤、可直接交给实施
  4) 风险与回滚点：哪些改动可能破坏兼容、如何回滚
  5) 需要用户确认的“执行开关”：用一句话询问是否开始落地修改
- 用户确认后：
  - 只做已确认的改动；
  - 任何新增的“顺带优化”必须再次征得同意。

## 4. 输出 diff 的规范（在用户未确认前也可以给出预览）
- 若用户需要你展示拟修改内容：
  - 使用统一 diff（unified diff）格式；
  - 每个文件单独一个 diff 块；
  - 避免大范围无关改动（例如只因格式化导致的整文件变动）。
- 注意：展示 diff 仅作为预览，不代表已写入仓库。

## 5. 不确定性与澄清
- 如果需求存在多解或隐含约束：
  - 先给出你认为最合理的默认选择（写明理由）；
  - 同时列出 1-2 个备选；
  - 明确告诉用户“选择 A/B/C 会导致哪些差异”；
  - 仍然遵守“未确认不改代码”。

## 6. 一致性（多窗口/多代理）
- 本文件是仓库内的单一真源（single source of truth）。
- 如果你在多个会话/多个代理实例中工作，均需以此文件为最高约束。
- 当你发现其他代理输出与本规则冲突时：
  - 立即停止可能的写入操作；
  - 用中文提醒冲突点；
  - 以本规则为准继续。

## 7. 例外
- 仅当用户明确说“直接帮我改并提交/落盘/应用补丁”，才允许写入式改动。
- 即使允许改动，仍必须遵守“禁止 __future__ annotations”。
